### Commit Message Rules

**Format:** `type(scope): description` (scope required)

**Valid types:** feat, fix, docs, style, refactor, perf, test, build, ci, chore, revert

**Limits:**

- Title: 50 chars max
- Body: 72 chars per line max

**NEVER:**

- Use feat/fix with infra scopes → ci(ci):, test(test):, docs(docs):, build(build):
- Include PR references → No #123 or github.com/owner/repo/pull/123
- Add AI attribution → No "Generated by Claude" or Co-Authored-By: Claude
- Omit scope → Must include (scope)

**Examples:**

- ✅ `feat(api): add user endpoint`
- ✅ `ci(actions): update workflow`
- ❌ `feat(ci): update workflow` (use ci(actions) instead)
- ❌ `fix(api): resolve #123` (no PR refs)
- ❌ `feat: add endpoint` (missing scope)

## Hooks

**Critical:** ALWAYS fix root cause when hooks fail. NEVER bypass.

**When hook fails:**

- Read error message carefully
- Fix underlying issue (syntax, linting, validation)
- Rerun same command - hook should pass
- Research proper fix if unclear

**NEVER:**

- Use --no-verify or similar bypass flags
- Work around hooks with alternative approaches
- Skip hook by changing workflow
- Disable hook validation

**Nix-specific:**

- darwin-rebuild hooks validate config - fix .nix files, don't skip build
- Pre-commit hooks enforce quality - fix code, don't bypass validation
- If nix flake check fails - fix syntax/references, don't override

## GitHub

- Your primary method for interacting with GitHub should be the GitHub CLI.
- When creating new repos, configure:
  - `gh api repos/{owner}/{repo} --method PATCH -f allow_squash_merge=true -f allow_merge_commit=true`
  - `--method PATCH -f allow_rebase_merge=false -f merge_commit_title=PR_TITLE -f merge_commit_message=PR_BODY`
  - Squash merge: enabled (default)
  - Merge commit: PR_TITLE + PR_BODY
  - Rebase: disabled

### Subissues

**Terminal:** `gh-add-subissue <parent-num> <child-num> [owner/repo]`

**Manual GraphQL (requires `GraphQL-Features: sub_issues` header):**
```bash
# Get issue IDs first
gh api graphql -f query='query($owner:String!,$repo:String!,$num:Int!){repository(owner:$owner,name:$repo){issue(number:$num){id}}}' -f owner=OWNER -f repo=REPO -F num=123

# Add subissue
gh api graphql -H "GraphQL-Features: sub_issues" -f query='mutation{addSubIssue(input:{issueId:"I_parent",subIssueId:"I_child"}){issue{number}subIssue{number}}}'
```

**When user requests adding subissues:** Use `gh-add-subissue` script (located in ~/.local/bin/)

### PR Validation Rules

**Note:** Currently disabled in klaudiush config

**Title requirements:**

- Format: `type(scope): description` (conventional commits)
- Max 50 chars
- Valid types: feat, fix, docs, style, refactor, perf, test, build, ci, chore, revert

**Body requirements (when enabled):**

- Must include `## Motivation` section
- Must include `## Implementation information` section
- Optional `## Supporting documentation` section
- Changelog line format: `> Changelog: type(scope): description` or `> Changelog: skip`
- No formal language: utilize, leverage, facilitate, implement
- Changelog must appear after Motivation section

**Forbidden patterns:**

- `\btmp/` or `\btmp\b` in title/body
- No `/tmp` directory references

**Examples:**

- ✅ Title: `feat(api): add user endpoint`
- ✅ Changelog: `> Changelog: feat(api): add user endpoint`
- ✅ Changelog: `> Changelog: skip` (for ci/test/chore)
- ❌ Title: `Add new endpoint` (missing type/scope)
- ❌ Body without Motivation section

**NEVER:**

- If still unclear after research, ASK what to do (don't skip/disable)

## Dynamic Mode Switching

### Exploration Mode

- Triggered by: research, understanding, unclear requirements
- Behavior: thorough analysis, ask questions, no assumptions
- Tools: Grep, Glob, Read, Task (Explore agent)

### Implementation Mode

- Triggered by: clear requirements, defined scope
- Behavior: complete working code, quality gates, testing
- Tools: Edit, Write, Bash

### Debugging Mode

- Triggered by: test failures, errors, unexpected behavior
- Behavior: systematic diagnosis, root cause analysis
- Tools: Read logs, extended thinking, git history

### Optimization Mode

- Triggered by: performance issues, technical debt
- Behavior: measure first, targeted improvements
- Tools: profiling, benchmarking, refactoring
